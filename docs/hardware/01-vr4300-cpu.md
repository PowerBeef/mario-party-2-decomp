# VR4300 CPU

The Nintendo 64 uses a **NEC VR4300**, a MIPS III–compatible 32-bit CPU. Mario Party 2 code is compiled for this target with GCC 2.7.2 (`-mips3 -mgp32 -mfp32` in the [Makefile](../../Makefile)).

## MIPS Subset Used by N64 Games

| Feature | N64 usage |
|---------|-----------|
| 32-bit GPRs (`$0`–`$31`) | All game and libultra code |
| Load/store, branch, `jal`/`jr` | Visible throughout [`asm/1060.s`](../../asm/1060.s) |
| COP0 (system control) | Exceptions, TLB, cache control — libultra only |
| COP1 (FPU) | Float math in game (`f32` in structs like `Vec`) |
| 64-bit `lld`/`scd` | Not used by MP2 |
| Hardware page tables | **None** — OS maps memory via **TLB entries** |

## Register Conventions (Reading MP2 Asm)

| Register | ABI name | Typical use in MP2 |
|----------|----------|-------------------|
| `$zero` | — | Always 0 |
| `$at` | `$1` | Assembler temporary |
| `$v0`–`$v1` | — | Return values |
| `$a0`–`$a3` | — | First four arguments |
| `$t0`–`$t9` | — | Temporaries (entrypoint BSS clear uses `$t0`–`$t2`) |
| `$s0`–`$s7` | — | Saved callee registers |
| `$gp` | `$28` | Global pointer for small data |
| `$sp` | `$29` | Stack pointer — entry sets `0x800FC730` |
| `$ra` | `$31` | Return address for `jal` |

Example from [`asm/entrypoint.s`](../../asm/entrypoint.s):

```mips
lui   $t0, %hi(D_800D4BF0)   # BSS start
addiu $t0, $t0, %lo(D_800D4BF0)
lui   $sp, %hi(D_800FC730)   # stack top
jr    $t2                     # jump to main
 addiu $sp, $sp, %lo(D_800FC730)  # delay slot: set SP
```

## Branch Delay Slots

MIPS executes the instruction **after** a branch/jump before transferring control. MP2 asm uses `.set noreorder` in [`include/macro.inc`](../../include/macro.inc), so every `jal`/`beq`/`jr` has an explicit delay-slot instruction. Misreading asm often means confusing the delay slot with unreachable code.

## Caches

| Cache | Size | Policy |
|-------|------|--------|
| I-cache | 16 KB | Read-only instructions |
| D-cache | 8 KB | Write-back |

The RCP **does not snoop** the CPU D-cache. When the CPU writes data the RSP/RDP/PI will read, software must **write back** or **invalidate** cache lines.

MP2 libultra functions in main (call counts from [call-inventory.md](call-inventory.md)):

| Function | VRAM | Calls | When |
|----------|------|-------|------|
| `osWritebackDCacheAll` | `0x800A1EF0` | 6 | Before RSP tasks |
| `osWritebackDCache` | `0x800A1E70` | 7 | Targeted buffer flush |
| `osInvalDCache` | `0x800A1D40` | 6 | After PI DMA into RDRAM |
| `osInvalICache` | `0x800A1DF0` | 4 | After loading new code |

Typical RSP submit region in main (~`0x8007E7B8`): `osWritebackDCacheAll` → `osSpTaskYield` / `osSpTaskStartGo`.

## TLB and Virtual Addresses

The VR4300 has **32 TLB entries** (dual-page). libultra installs mappings so:

- RDRAM appears at **`0x80000000`** (cached KSEG0)
- Cartridge ROM appears at **`0x10000000`** region during access
- Uncached aliases exist at **`0xA0000000`** (KSEG1)

`osVirtualToPhysical` @ `0x800A2470` is called **46 times** in main — almost always before DMA so the PI/RSP sees a **physical RDRAM address**, not a TLB handle.

## Exceptions and libultra Threads

Hardware exceptions (TLB miss, interrupt, syscall) enter through **`0x80000080`** (general exception vector in KSEG0). libultra installs handlers that:

1. Save CPU state
2. Dispatch to interrupt service or thread scheduler
3. Restore and `eret`

**HuPrc processes** (`InitProcess`, `SleepProcess` in engine docs) are **not** separate CPU threads. They are coroutines using `setjmp`/`longjmp` on the **same** VR4300. True parallelism is only:

- libultra OS threads (VI manager, PI manager, SI manager, idle)
- RSP microcode running concurrently with CPU

See [../03-process-system.md](../03-process-system.md) for the engine scheduler; this doc covers the silicon underneath.

## COP0 Registers (Reference)

| Register | Purpose |
|----------|---------|
| `$12` Status | Interrupt enable, user/kernel mode |
| `$13` Cause | Exception type, pending IP bits |
| `$15` PRId | Processor ID |
| Index/EntryHi/EntryLo | TLB programming |

Game code rarely touches COP0 directly; libultra wraps TLB and exception setup during `osInitialize`.
